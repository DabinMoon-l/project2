## 3. 로컬 분석 환경 적합성

| 단계 | 데이터 크기 | 노트북 처리 |
|---|---|---|
| 원시 pageViews 로드 | ≈15만 행 × 7열 (≈15 MB CSV) | ✅ 여유. 단일 데이터프레임(R `data.frame`/`data.table`), 수 초 내 |
| 집계(학생×변수) | groupby ≈148 그룹 | ✅ 즉시 |
| 최종 wide CSV | ≈148 × 40 | ✅ 메모리 무시 가능 수준 |
| 통계 분석(J-T 추세 + 효과크기·95%CI) | ≈148행 | ✅ R `DescTools`·`effsize`·`boot` 즉시 |

**결론: 일반 노트북(RAM 8GB+) + R로 전 과정 처리 가능.** 서버/클러스터 불필요.

### 권장 처리 순서(메모리 절약)
1. 컬렉션별로 추출 → 컬렉션별 중간 CSV 저장(원시 ID 포함, **비식별 전 임시·즉시 삭제 대상**).
2. 컬렉션별로 학생 단위 집계 → 변수 partial CSV.
3. partial 병합 → wide CSV(비식별 완료본). 임시 파일 폐기.
4. 이후 분석은 wide CSV(<1MB)만 사용 → 재현·공유 용이.

## 4. 리스크 메모

- pageViews 추정치 변동 큼(학생별 사용 편차). 실측이 30만 행을 넘어도 30 MB 수준 → 여전히 노트북 OK.
- Firestore 읽기 비용: 추출은 1회성·종강 후 → 운영 비용 영향 미미. 그래도 날짜·courseId 필터 필수.
- RTDB(`tekken/results`)는 별도 export 필요(Firestore와 다른 경로) — 추출 스크립트에서 분기 처리(설계만).
- ⚠️ `comments` raw가 비공개 콩콩이·gemini-ai 답변·공동연구자(PI ~1,000개)로 크게 부풀려져 있음(초기 추정 3~5천은 *분석 대상* 기준이지 raw가 아님). raw 합계가 20~30만 행대로 커져도 pageViews가 여전히 지배적이고 총 용량 수십 MB라 노트북 처리 결론 불변. **단 집계 전 PI 제외·`isPrivate=false`·학생 authorId 필터를 반드시 raw 스캔 직후 적용**(어떤 학생 단위 분모에도 PI·AI·비공개가 새지 않도록).
